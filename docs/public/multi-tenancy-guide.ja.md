# マルチテナンシーガイド

Searchable は **Namespace** を通じてマルチテナンシーをサポートします。
各テナントまたはデータセットは、独立した Analyzer、埋め込み、永続化
設定を持つ独自の論理インデックスを持ちます。本ガイドでは、Namespace
が実際に何を分離するか、そして、より重要な点として何を分離 **しない**
かを説明します。これにより、信頼できないテナントに公開する前に、
Namespace が自身のテナンシーモデルに適合するかどうかを判断できます。

## 分離モデル

| 境界 | 分離される？ | 備考 |
| --- | --- | --- |
| Lucene インデックスファイル | はい | 各 Namespace は設定されたインデックスルート配下に個別のディレクトリを持つ |
| Analyzer / 埋め込み戦略 | はい | Namespace 単位の `searchable.yaml` 設定 |
| メタデータ行（HikariCP / H2 / PostgreSQL） | はい（論理的に） | 行は `namespace_id` でタグ付けされ、スキーマは共有 |
| JVM ヒープ / スレッドプール | **いいえ** | すべての Namespace が 1 つのプロセス、1 つのヒープ、1 つの executor を共有 |
| ファイルディスクリプター / `MMapDirectory` マッピング | **いいえ** | Namespace をまたいだオープン中の searcher が同じ FD バジェットを奪い合う |
| CPU / ディスク I/O | **いいえ** | Namespace 間の QoS やレート制限はなし |
| 保存時暗号化 | **いいえ** | インデックスファイルとメタデータは平文で保存される |
| 認証 / 認可 | **いいえ** | ライブラリは呼び出し元を認証しない。アクセス制御は組み込み先アプリケーションの責任 |
| `AiProvider` からの送信トラフィック | **いいえ** | 外部プロバイダー（Anthropic / OpenAI）が有効な場合、取得したヒットがホストから流出する。下記§5 を参照 |

要するに、Namespace は 1 つの JVM 内における **論理的な** 境界であり、
プロセス、コンテナー、クラスターレベルの分離ではありません。

## 計画時に考慮すべき制約

### 1. OOM リスクはテナント間で共有される

Lucene の `IndexWriter`、セグメントマージ、HNSW ベクトルインデックス
は共有された JVM ヒープ上で割り当てを行います。暴走するセグメント
マージを起こした単一の Namespace がプロセス全体を `OutOfMemoryError`
に追い込み、他のすべての Namespace を巻き添えにする可能性があります。

- テナント供給データを受け入れる場合は、Namespace 単位のドキュメント
  数またはインデックスサイズをアプリケーション層で上限設定してください。
- `-Xmx` は最大の Namespace 単体ではなく、想定される Namespace の
  **合計** に余裕を持たせて設定してください。
- 明示的な `forceMerge` は、他の Namespace が高トラフィッククエリを
  処理していないときにのみ実行してください。

### 2. ノイジーネイバー効果（QoS なし）

Searchable は Namespace 間でレート制限、クエリコスト計算、優先度
キューを適用 **しません**。高コストなフレーズクエリや大きな `k` での
kNN 検索を実行する単一の Namespace は、同じプロセス上の他のすべての
Namespace のレイテンシーを増加させます。

- 公平性が必要な場合、価値の高いテナントごとに 1 プロセスを起動し、
  それぞれを自身の `searchable.yaml` に向けてください。
- 共有デプロイメントでは、テナント単位のリクエストバジェットと
  クエリタイムアウトを組み込み先アプリケーションで強制してください。

### 3. 保存時暗号化なし

Lucene のセグメントファイル、ベクトルファイル、メタデータ DB は
平文で書き込まれます。

- 脅威モデルが保存時の機密性を要求する場合、暗号化ファイルシステム
  （LUKS、dm-crypt、APFS 暗号化ボリューム）または暗号化ブロック
  デバイス（KMS による EBS gp3、CMEK による GCE PD など）上に
  デプロイしてください。
- テナント単位の暗号化（Namespace ごとに異なる鍵）は利用できません。
  これは `BACKLOG-002` で追跡されています。

### 4. 組み込みの認証 / 認可なし

Java API はすべての呼び出し元を信頼します。Namespace は **データ**
のみを分離し、**アクセス** は分離しません。以下は組み込み先
アプリケーションの責任です。

- 呼び出し元を認証する（OIDC / API キー / mTLS など）
- 各呼び出し元がどの Namespace を読み書きできるかを認可する
- Namespace 単位のクエリおよび ingest バジェットを強制する

[`examples/`](../../examples/) 配下のリファレンス REST API と MCP
サーバーは 1 つの配線例を示していますが、これらは説明用であり、
敵対的なテナントに対して堅牢化されてはいません。

### 5. 外部 AI プロバイダーはテナントデータをホスト外へ送信する

`searchable-ai` は検索後の要約 / Q&A のために 3 つの `AiProvider`
実装を同梱します。

| プロバイダー | エンドポイント | データの流れ |
| --- | --- | --- |
| `AnthropicProvider` | `https://api.anthropic.com` | クエリと取得したヒットを Anthropic に送信 |
| `OpenAiProvider` | `https://api.openai.com/v1` | クエリと取得したヒットを OpenAI に送信 |
| `OllamaProvider` | `http://localhost:11434`（設定可能） | ホスト内（または Ollama デーモンが動作する場所）に留まる |

デフォルトではどのプロバイダーも選択されません。**Anthropic または
OpenAI を有効化することは、呼び出し元 Namespace のドキュメント内容
がプロバイダーのデータ保持・学習利用ポリシーに従って、HTTPS 経由で
平文のままホストから流出することを意味します**。マルチテナント
デプロイメントでは、これは Searchable 自身の分離モデルを超える影響
を持ちます。

- **データレジデンシー / 規制範囲**（GDPR、個人情報保護法、HIPAA、
  金融業界規則）— テナントのドキュメントを米国拠点の LLM プロバイダー
  に送信することは、契約上または法令上のレジデンシー要件に違反する
  可能性があります。規制対象テナントに対して有効化する前に、選択
  したプロバイダーのリージョン、ゼロ保持オプション、DPA を確認して
  ください。
- **テナント単位のオプトインは呼び出し元アプリケーションの仕事です**
  — ライブラリには `AiProvider` の利用を許可 / 拒否するための
  Namespace 単位のフラグはありません。組み込み先アプリケーションは
  リクエストごとに、呼び出し元のデータを設定されたプロバイダーと
  共有してよいかを判断しなければなりません。
- **機密性の高いテナントには Ollama（または他の自己ホスト型
  プロバイダー）を優先してください** — Ollama はプロンプトをホスト
  内に保持します。外部プロバイダーが受け入れられない場合、自己ホスト
  型推論エンドポイントに対するカスタム `AiProvider` の実装が標準的
  な経路です。

## Namespace で十分な場合

- 同じプロセス内に複数の **データセット**（例：「support」、
  「engineering」、「marketing」）を持つシングルテナントデプロイメント。
- すべてのテナントが内部関係者であり、性能分離が契約要件ではない、
  信頼できるマルチテナントデプロイメント。
- 開発者または機能ブランチごとに 1 つのインデックスプロセスを持ち
  たい開発 / ステージング環境。

## Namespace では不十分な場合

- 1 つのテナントが他のテナントから CPU、メモリ、レイテンシーを奪う
  ことが許されない、信頼できないマルチテナント SaaS。
- テナント単位の暗号化鍵を必要とするワークロード（HIPAA / PCI DSS /
  地域データレジデンシー）。
- QoS 保証を伴うテナント単位の厳格な SLA。

これらのシナリオでは、テナント（またはテナントコホート）ごとに
1 つの Searchable プロセスを起動し、ルーティング層を Searchable の
上位に置いてください。
