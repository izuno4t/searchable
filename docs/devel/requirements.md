# Searchable - 要求仕様書

## 1. プロジェクト概要

### 1.1 目的

軽量で高速な全文検索・ベクトル検索機能を提供するJavaライブラリ。**日本語を主要対象言語**とし、Webアプリケーションへの組み込み、REST API経由での利用、MCPツールとしての利用など、多様なユースケースに対応する。

### 1.2 主要な特徴

- **日本語最適化**: 日本語形態素解析と日本語特化の埋め込みモデル
- **ハイブリッド検索**: 全文検索とベクトル検索を単独または組み合わせて利用可能
- **マルチテナント対応**: Namespaceによる論理的な分離
- **マルチAPI**: Java API、REST API、MCPサーバーとして提供
- **高速レスポンス**: インメモリアーキテクチャによる高速検索
- **プラグイン拡張**: データソースや検索処理のカスタマイズ

### 1.3 主なユースケース

- Webアプリケーションへの簡易検索機能の組み込み
- MCPツール用のドキュメント検索サーバー
- 複数のドキュメントソースの統合検索管理

## 2. 機能要件

### 2.1 検索機能

#### 2.1.1 全文検索

- キーワードベースの高速全文検索
- **日本語形態素解析**（Kuromoji/Sudachiによる分かち書き）
- **日本語の適切なトークン化**（助詞、助動詞の処理）
- マルチバイト文字対応（UTF-8）
- フィールド指定検索
- ファジー検索、ワイルドカード検索
- ハイライト機能（日本語テキスト対応）
- **コンテンツ重み付け**
  - 見出しレベルに応じた自動ブースト（h1=7.0, h2=6.0, ..., h6=2.0,
    その他=1.0）
  - 任意セクションへのカスタム重み設定（0.0〜10.0）
  - 二次関数スケーリング（weight 2.0 → 約4倍のインパクト）
- **ユーザー辞書（カスタム分かち書き）**
  - Kuromoji の UserDictionary 形式
    （`surface,segmentation,reading,pos`）を読み込み可能
  - スコープ階層: グローバル辞書（全Namespace適用）+
    Namespace 個別辞書（グローバルを継承し、追加・上書き）
  - ストレージは選択可能（ファイル / メタデータDB）
  - 同一の表層形は Namespace 個別エントリがグローバルを上書き
  - 設定変更後の新規インデックス・再構築時に反映

#### 2.1.2 ベクトル検索（MVP範囲）

- Javaのみで完結する埋め込み処理（Onnx Runtime使用）
- **日本語対応埋め込みモデル**（multilingual-e5-small/intfloat等）
- セマンティック類似度検索
- レスポンス性能: 500ms以内（単一Namespace）

#### 2.1.3 ハイブリッド検索

- 全文検索とベクトル検索の組み合わせ
- 検索順序の設定
  - シーケンシャル: 全文検索 → ベクトル検索 または逆順
  - パラレル: 両方同時実行してマージ
- AI統合（オプション）: 複数の検索結果をAIで要約・統合

#### 2.1.4 検索結果

- ページネーション対応
- スコアリング
- メタデータ付与（予約キー: `url`(オリジン URI)・`contentType`(MIME)・`category`・`lang`・`tags` ─ 詳細は `docs/devel/design/architecture/overview.md` §5.7）
- **セクション単位の検索結果（Sub-results）**
  - ドキュメント内の見出し単位で結果を返却
  - 各セクションへのアンカー付き URL（`metadata.url + #heading-slug`、full-text 検索でのみ生成）
  - セクションごとの個別スコアリングとエクセプト
- **スニペット自動生成**
  - ヒット箇所周辺のテキスト抜粋を自動生成
  - 検索語のマークアップ付き（`<mark>`タグ等）
  - エクセプト長の設定（デフォルト文字数の指定）
  - HTMLエンコード済みスニペットとプレーンテキスト版の両方を提供
- **ファセット検索（Faceted Search）**
  - フィルタ値と該当件数の集計API
  - メタデータ属性によるフィルタリング
  - 複数値フィルタ（AND/OR）
  - 予約済みフィルタキー（any, all, none, not）の定義
  - インライン指定・属性値指定・要素内容指定の3方式
- **ランキングチューニング**
  - 以下のパラメータをAPI経由で設定可能:
  - BM25パラメータのカスタマイズ
  - Namespace/検索リクエスト単位での上書き

| パラメータ | 説明 | デフォルト |
| --- | --- | --- |
| termFrequency | 語の出現頻度の影響度 | 1.0 |
| termSimilarity | 検索語との一致度の影響 | 1.0 |
| pageLength | 短いドキュメントの優遇度 | 0.75 |
| termSaturation | 語の繰り返しの飽和点 | 1.4 |
| metaWeights | メタデータフィールド別の重み | title=5.0 |

- **遅延ロード結果（Lazy-loaded Results）**
  - 検索時はID・スコア・URLのみを返却（軽量レスポンス）
  - 詳細データ（本文、メタデータ等）はオンデマンド取得API
  - 大量結果時のパフォーマンス最適化

### 2.2 Namespace機能

#### 2.2.1 基本機能

- 論理的なインデックス分離
- 動的な作成・削除（API経由）
- Namespaceごとの検索実行
- 複数Namespace横断検索
- **Namespace横断検索の重み付け**
  - Namespaceごとの検索重み（indexWeight）設定
  - 高い重みのNamespaceの結果を優先的にランキング
  - マージフィルタによるNamespace別の結果分類

#### 2.2.2 設定管理

- グローバルデフォルト設定
- Namespace個別設定（個別設定が優先）
- 設定可能項目:
  - 検索アーキテクチャ（全文検索/ベクトル検索/ハイブリッド）
  - 検索順序
  - AI統合の有効/無効
  - 埋め込みモデルの選択
  - その他検索パラメータ

#### 2.2.3 権限管理（設計のみ、実装は将来）

- ユーザー/ロール管理を見据えた設計
- MVPでは権限チェックなし（組み込み側で制御）

### 2.3 インデックス管理

#### 2.3.1 ドキュメント登録

対応フォーマット:

- **MVP範囲**:
  - テキスト系: Plain Text, Markdown, AsciiDoc
  - PDF
  - HTML
- **将来対応**:
  - Office系: Word, Excel, PowerPoint
  - Google Docs, Apple Pages等（PDF変換での対応を推奨）

#### 2.3.2 インデックス更新

- ドキュメント変更の検知
- 非同期インデックス更新
- 差分更新対応
- バッチ更新API

#### 2.3.3 永続化

ドキュメント・インデックス・メタデータは抽象化されたストレージに配置し、
バックエンドを選択可能とする。

- **Lucene インデックスのバックエンド**
  - ローカルファイルシステム (既定)
  - インメモリ (テスト用)
  - オブジェクトストレージ (S3 互換)
- **メタデータ DB のバックエンド**
  - H2 (ファイル / インメモリ)
  - RDB (TCP 経由 JDBC、例: PostgreSQL、H2 サーバーモード)
- **元ドキュメントの配置先**
  - ローカルファイルシステム
  - オブジェクトストレージ (S3 互換)

#### 2.3.4 バックアップ・リストア

- 自動バックアップ機能
- 手動エクスポート/インポートAPI
- バックアップスケジュール設定

#### 2.3.5 同時実行制約

- **書込(インデックス更新)は Namespace 単位で1プロセスのみ**
  - Lucene の `write.lock` による制約。ロックは Namespace の
    インデックスディレクトリ単位で取得される
  - 同一 Namespace に対する `searchable-cli` と `searchable-ui`
    (設定・運用 Web) の同時書込起動は不可
  - 異なる Namespace なら複数プロセス・複数モジュールから並列更新可能
    (例: Namespace A は CLI、Namespace B は設定 Web)
  - 同一プロセス内の複数リクエストはライブラリ側でシリアライズ
- **参照(検索)は複数プロセスで並列実行可能**
  - `searchable-api` / `searchable-mcp` / `searchable-search-ui` は
    読込専用モードで同一ストレージを共有できる
  - 参照は Namespace 横断・複数プロセス同時に可能

### 2.4 API提供

#### 2.4.1 Java API

- ライブラリJARとして配布
- 主要機能:
  - 検索実行
  - インデックス管理
  - Namespace管理
  - 設定管理

#### 2.4.2 利用パターン別サンプル

代表的な統合パターン (普通の Web アプリ / API 経由 / MCP 経由) ごとに
サンプルアプリを `examples/` 配下に提供する。一覧と各サンプル要件への
リンクは [examples/README.md](../examples/README.md) を参照。

ライブラリ本体としては、これらサンプルが必要とする「読込専用モード」や
「API Key 認証フック」など共通の API/拡張点を提供することのみを規定する。

#### 2.4.3 CLI（コマンドラインツール）

- `searchable-cli` モジュールとして提供
- インデックス管理コマンド全般を提供:
  - 取込(単一・バッチ、データソースプラグイン経由含む)
  - 削除(ドキュメント単位)
  - 再構築
  - バックアップ・リストア
  - インデックス状態確認
  - 設定検証(ドライラン)
- サーバー起動なしで運用作業を実行可能

### 2.5 設定・運用 Web (`searchable-admin`)

設定と運用作業のための Web アプリケーション。Thymeleaf ベース、優先実装。
**書込側プロセス** として稼働し、CLI とは同一 Namespace に対して同時に
書込モードで起動できない(2.3.5 参照)。

- Namespace 管理
  - 一覧表示、作成、編集、削除
  - 設定変更(ドキュメントパス、インデックスパス含む)
- インデックス管理
  - インデックス状態確認
  - 手動インデックス更新(オンデマンド実行)
  - インデックス再構築
  - インデックスの部分更新・削除
  - ドキュメント登録/削除
- ユーザー辞書管理
  - グローバル辞書 / Namespace 個別辞書
  - ファイル/DB ストレージの切替
- ランキング設定
  - BM25 パラメータ、metaWeights、indexWeight
- AI 統合設定
  - 利用 AI サービスの選択(OpenAI / Anthropic / Ollama)
  - API キー・モデル設定
- バックアップ設定
  - 自動バックアップスケジュール
  - 手動エクスポート/インポート
- モニタリング
  - インデックスサイズ
  - 検索パフォーマンス統計

### 2.6 サンプルアプリ

利用パターン別のサンプルは 2.4.2 を参照(`examples/` 配下に個別配置)。

### 2.7 プラグイン機構

#### 2.7.1 MVP範囲

- **データソースプラグイン**
  - カスタムデータソースからのドキュメント取得
  - プラグインインターフェースの定義
  - JARファイル配置による読み込み

#### 2.7.2 将来拡張

- カスタム検索フィルタ
- 文書パーサープラグイン
- カスタムスコアリング

### 2.8 AI統合（オプション機能）

**`searchable-ai` 独立モジュール** として提供。利用側
(`searchable-admin` / 各種サンプル等) は API を通じて呼び出すのみで、
core 本体には AI 依存を持ち込まない。

#### 2.8.1 対応AI API

- OpenAI API
- Anthropic Claude API
- ローカルLLM（Ollama等）
- 複数AI APIの切り替え対応(設定で切替)

#### 2.8.2 AI機能

- 検索結果の要約・統合(検索後処理)
- セマンティック検索の拡張
- 設定によるON/OFF切り替え

**注**: ベクトル検索用の埋め込み生成 (ONNX Runtime) は
2.1.2 で別建てとし、`searchable-core` 内に閉じる。本節の AI 統合は
あくまで「外部 AI サービスを呼び出す要約・統合の後処理」を指す。

#### 2.8.3 制御

- タイムアウト設定（調査中）
- 費用制限はAIサービス側で管理

## 3. 非機能要件

### 3.1 性能要件

#### 3.1.1 検索レスポンス

- **目標**: 500ms以内（単一Namespace検索時）
- **想定文書数**: Namespaceあたり最大100,000件
- **同時接続数**: 実行環境依存（アプリケーション側での制約なし）

#### 3.1.2 インデックスサイズ

- メモリベース: 数GB程度まで対応
- ディスクベース: 実行環境依存で制限なし

#### 3.1.3 スケーラビリティ

- Namespace単位での分散可能な設計
- 水平スケーリングは実行環境に依存

### 3.2 可用性・信頼性

- インデックスの永続化による再起動後の復旧
- 自動バックアップ機能
- エラーハンドリングとログ出力

### 3.3 保守性

- SLF4J + Logbackによる構造化ログ
- 設定ファイルによる柔軟な構成管理
- プラグイン機構による拡張性

### 3.4 セキュリティ

#### 3.4.1 認証・認可

- **API Key 認証**(REST API および MCP サンプル)
  - 環境変数または設定ファイルで API Key を指定したとき有効化
  - 未設定時は認証なし(開発・組み込み利用向け)
- Bearer Token 認証は採用しない(MVP/Phase 1 範囲)
- 将来的なユーザー/ロール管理を見据えた設計

#### 3.4.2 データ保護

- インデックスデータの暗号化（将来検討）
- 通信の暗号化（HTTPS、組み込み側で対応）

## 4. 制約事項・前提条件

### 4.1 実行環境

- Java 21以上
- メモリ: Namespaceあたり数GB推奨（インメモリモード時）
- ディスク: インデックスサイズに応じた容量

### 4.2 ライセンス

- 使用するOSSライブラリのライセンス確認
- ライブラリ配布時のライセンス明記

### 4.3 サポート範囲

- 日本語を含むマルチバイト文字対応
- UTF-8エンコーディング

## 5. 用語集

- **Namespace**: 論理的なインデックス分離単位
- **埋め込み（Embedding）**: テキストをベクトル表現に変換すること
- **ハイブリッド検索**: 全文検索とベクトル検索の組み合わせ
- **MCP**: Model Context Protocol、AIツールとの連携プロトコル
