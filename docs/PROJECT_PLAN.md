## 2. 技術調査項目

### 2.1 Phase 1 開始前（優先度: 最高）

#### 全文検索エンジン選定
- **調査内容**:
  - Apache Luceneの機能・性能評価
  - 日本語形態素解析器の選定（Kuromoji vs Sudachi）
  - 日本語検索精度の検証
  - ストップワード、同義語辞書の整備
- **期限**: Phase 1開始前1週間
- **成果物**: 選定レポート、簡易PoC（日本語サンプルデータ使用）

#### 軽量DB選定
- **調査内容**:
  - H2, SQLite, RocksDBの性能比較
  - インデックスサイズとメモリ使用量の評価
  - 永続化方式の最適化
- **期限**: Phase 1開始前3日
- **成果物**: 選定レポート、ベンチマーク結果

#### 性能目標の検証
- **調査内容**:
  - 100,000件規模での検索レスポンス測定
  - 目標500ms以内の達成可能性
- **期限**: Phase 1開始前
- **成果物**: 性能測定レポート

### 2.2 Phase 1 期間中（Phase 2 準備）

#### ベクトル検索の実現可能性
- **調査内容**:
  - Onnx Runtime Javaの組み込み検証
  - 日本語対応埋め込みモデルの評価
    - multilingual-e5-small
    - multilingual-e5-base
    - paraphrase-multilingual-MiniLM-L12-v2
  - 日本語テキストでのベクトル化精度検証
  - メモリ使用量の測定
- **期限**: Phase 1終了まで
- **成果物**: PoCコード、性能評価レポート（日本語検索精度含む）

#### ベクトル検索エンジン選定
- **調査内容**:
  - Lucene HNSWの評価
  - 代替ライブラリの調査
  - 100,000件規模でのレスポンス測定（目標: 500ms以内）
- **期限**: Phase 1終了まで
- **成果物**: 選定レポート、ベンチマーク結果

#### 非同期処理方式
- **調査内容**:
  - インデックス更新の非同期処理設計
  - 変更検知の仕組み（ファイルウォッチャー等）
  - Java 21のVirtual Threads活用検討
- **期限**: Phase 1終了まで
- **成果物**: 設計ドキュメント

### 2.3 調査の進め方

#### タイムボックス方式
- 各調査項目に明確な期限を設定
- 期限内に結論を出し、必要に応じて代替案を選択

#### 優先順位
1. **最高優先**: Phase 1開始に必須の調査（検索エンジン、DB選定）
2. **高優先**: Phase 2に必須の調査（ベクトル検索）
3. **中優先**: 最適化・改善に関する調査

#### 判断基準
- 機能要件を満たすか
- 性能目標を達成できるか
- 導入・保守の容易さ
- ライセンスの適合性
- コミュニティの活発さ

## 4. リスク管理

### 4.1 Phase 1 のリスク

#### リスク: 検索エンジン選定に時間を要する
- **影響度**: 高
- **対策**: 
  - Luceneを第一候補として早期に検証開始
  - 1週間をタイムボックスとして設定
- **代替案**: Apache Luceneに決定

#### リスク: 性能目標（500ms）が達成できない
- **影響度**: 中
- **対策**: 
  - 早期に性能測定を実施
  - キャッシュ戦略の検討
  - インデックス最適化
- **代替案**: 目標値を緩和（1秒以内）

#### リスク: 軽量DB選定が難航する
- **影響度**: 中
- **対策**: 
  - H2を第一候補として検証
  - 3日をタイムボックスとして設定
- **代替案**: ファイルシステムベースの実装を優先

### 4.2 Phase 2 のリスク

#### リスク: ベクトル検索の性能が目標に達しない
- **影響度**: 高
- **対策**: 
  - Phase 1期間中にPoCを実施
  - 性能測定を優先実施
  - メモリ使用量の最適化
- **代替案**: 
  - Phase 2の範囲を縮小（ベクトル検索のみ）
  - 外部APIベースに変更してPhase 2へ延期

#### リスク: Onnx Runtime統合が複雑
- **影響度**: 中
- **対策**: 
  - 公式ドキュメント・サンプルの事前調査
  - シンプルなPoCで統合確認
- **代替案**: 別のJava用埋め込みライブラリを検討

#### リスク: メモリ使用量が想定を超える
- **影響度**: 中
- **対策**: 
  - メモリプロファイリングの実施
  - ディスクベースの最適化
- **代替案**: 完全ディスクベースに切り替え

### 4.3 Phase 3 のリスク

#### リスク: UI実装に想定以上の時間がかかる
- **影響度**: 低
- **対策**: 
  - シンプルなデザインに留める
  - Bootstrap等のCSSフレームワーク活用
- **代替案**: 最小限の機能のみ実装

### 4.4 全体的なリスク

#### リスク: スケジュール遅延
- **影響度**: 中
- **対策**: 
  - 各フェーズの終わりで進捗確認
  - スコープ調整の柔軟な判断
  - 技術調査にタイムボックスを設定

#### リスク: 想定外の技術課題
- **影響度**: 中
- **対策**: 
  - 早期の技術検証（PoC）
  - 定期的な技術レビュー
  - 代替技術の候補をリストアップ
